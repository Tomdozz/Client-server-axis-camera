Is the timestamp received with the capture library the same as in the JPEG header?

Yes, but in a different format.

Can I find the raw image that corresponds to the JPEG image?

The timestamp received together with the uncompressed image buffer can be compared with the timestamp of the JPEG image. If the timestamps are equal both images come from the same sensor image data, but there is no guarantee that there will be any equal timestamp.

How do I get information about the product my application is running on?

Read the following VAPIX® parameters using the parameter interface: 

root.Brand.ProdShortName
root.Properties.System.HardwareID
root.Properties.Architecture
root.Properties.EmbeddedDevelopment.Version

What if one of the used interfaces is updated so that my application is no longer compatible?

The interfaces have a major and a minor version. For example, if the directory /usr/lib in the Axis product contains a file libcapture.so.1.0, then the capture library has major version 1 and minor version 0. If the application is compiled and linked against the same major version as in the product, the application should work fine.

If the major version does not match the major version of the library used to build the application, installing the application in the product will fail and an error will be written to the log. Upgrading the product firmware to a version that includes interfaces that are not compatible with the ones used to build the application will result in a similar error.

To avoid this, make sure that the major version of the library in the SDK is the same as the major version of the library in the product. To ensure this, always use the latest version of the SDK and of the product firmware. When creating an application using the interfaces introduced in SDK 2.0, remember to set REQEMBDEVVERSION="2.0" in package.conf . This correspond to firmware version 5.60.

What happens with installed applications when performing a firmware upgrade?

During a firmware upgrade, installed applications and their parameters are preserved and will not be affected as long as the new firmware contains compatible API:s.

What happens with installed applications when performing a factory default?

During restore (soft factory default), installed applications and their parameters are perserved. To restore the product, go to the Setup > System Options > Maintenance page and click Restore.

During factory default, installed applications and their parameters are removed. To reset the product to factory default, got to the Setup > System Options > Maintenance page and click Default. If the web pages are not accessible, it is also possible to reset the product using the control button. Refer to the product's Installation Guide or User Manual for instructions.

What happens "under the hood" when an application package is installed?

When an application package is uploaded to the Axis product using the eap-install.sh script or using the product's web pages, a cgi that unpacks the package file is executed. The cgi then reads the package.conf file and creates a directory called /usr/local/packages/APPNAME, where APPNAME is defined in package.conf. The cgi also creates symbolic lnks from the /etc directory (as this is the location that is searched for configuration files by various daemons) to the new application directory, which contains the actual configuration files. Finally, the cgi creates an init script (/etc/init.d) that makes sure the application is started when the Axis products is started.

Why do I get an error code when I use the filters in the rapp library?

The filters in the rapp library will look outside the image data for padding pixels to calculate the correct value for the edge pixels. If a frame is used unmodified in a filter, there is no padding pixels for rapp use for the calculation and therefore rapp gives an error message. Use a subset of the image for filtering or allocate a bigger buffer and copy the image to the new buffer and call the padding functions.

Running the application in host environment

In some cases, it may be easier to first develop an application on the host system and then upload it to the Axis product. Note that development on host is limited to the capture, fixmath and rapp libraries.

  • Using the capture library on host
  • Using the fixmath library on host
  • Using the rapp library on host

Using the capture library on host

The host version of the capture library can open a JPEG stream from the Axis product or can work with a MJPEG file as input. If the media type is IMAGE_UNCOMPRESSED the capture host library convert the JPEG frame to the Y800/I420 format. For more information, see capture page.

To build an application for host, update Makefile with support for host building: 

ifeq ($(AXIS_BUILDTYPE),host)
LDFLAGS += -lcapturehost -ljpeg
else
LDFLAGS += -lcapture
endif

Build the application: 

user@host$ make host
Updating .target-makefrag
user@host$ make clean
user@host$ make

Using the fixmath library on host

The fixmath library is supplied as source code in the SDK and is located under /libs/fixmath. For more information, see Appendix 1: Analytics libraries.

To build the fixmath library: 

user@host$ source init_env
user@host$ cd libs/fixmath
user@host$ make
user@host$ make install

Update Makefile with the fixmath library: 

LDFLAGS += -lfixmath

Using the rapp library on host

The rapp library is supplied as source code in the SDK and is located under /libs/rapp. For more information, se Appendix 1: Analytics libraries. To build the rapp library, libtool and autoconf must be installed as described in Prerequisites.

To build the rapp library: 

user@host$ source init_env
user@host$ cd libs/rapp
user@host$ make
user@host$ make install

Update Makefile with the rapp library: 

LDFLAGS += -lrapp

The rapp library will be installed under target/host/lib and the include files will be added to target/host/include.
